# Attention Terminal — Demo Script & Narration Guide

> **Target duration**: 3:00–3:30  
> **Format**: live screen recording, no slides, no intro logos  
> **Goal**: show the product doing useful work in a way a reviewer can follow quickly

---

## Demo flow

| Time | On screen | Narration |
| --- | --- | --- |
| 0:00–0:45 | Open the homepage and ask a question in chat. | “Attention Terminal turns developer activity into visual answers instead of long text responses.” |
| 0:45–1:30 | Show the first rendered response and the morphing canvas. | “The response is a card or chart, not a paragraph. Each answer is tuned to the shape of the data.” |
| 1:30–2:15 | Open a repository drill-down card. | “This view combines repo metadata, recent activity, and the most relevant events in one place.” |
| 2:15–2:45 | Show the ClickHouse / Trigger.dev workflow or a query/transformation example. | “ClickHouse handles the live analytics, while Trigger.dev runs ingestion and background jobs.” |
| 2:45–3:15 | End on the architecture / telemetry / review flow. | “The system is designed to be inspectable, measurable, and easy to extend.” |

---

## Suggested narration

### 1. Opening
“This is Attention Terminal. The goal is simple: turn developer activity into visual answers that are easier to scan than a wall of text.”

### 2. Visual response
“When I ask a question, the app chooses a chart or card that matches the data. The result is immediately readable and stays focused on the question.”

### 3. Repo drill-down
“A repository view brings together summary metrics, recent events, and deeper activity details. You can inspect the data without leaving the page.”

### 4. Engine and architecture
“The product is built on ClickHouse for fast analytics and Trigger.dev for background work and ingestion. That lets the UI stay responsive while the data layer does the heavier lifting.”

### 5. Close
“The point of the project is not just to answer questions, but to answer them in a form that is fast to read and easy to verify.”

---

## Recording checklist

- Use a clean browser window and dark mode.
- Keep the cursor movement slow and deliberate.
- Avoid pausing on technical details longer than needed.
- Show the live product first, then the supporting infrastructure.
- Keep the final recording inside the requested time limit.
